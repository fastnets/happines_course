# Ограничения и правила

## Архитектурные
- Сохранять текущий стиль проекта: сервисы + репозитории + handlers.
- Не делать широкий рефакторинг без отдельной задачи.
- Любые изменения в расписании/начислениях делать с учетом идемпотентности.

## Качество
- Для критичных изменений добавлять или обновлять unit-тесты.
- Перед коммитом запускать:
  - `python -m unittest discover -s tests -p "test_*.py" -v`
- Не оставлять мертвый код и неиспользуемые модули.

## Безопасность
- Не коммитить секреты (`.env`, ключи, токены).
- Поддерживать `.env.example` актуальным и без чувствительных значений.
- Не выполнять деструктивные git-команды без явного запроса.

## Операционные
- Изменения должны быть минимальными и целевыми.
- Каждый логически отдельный блок изменений — отдельный коммит.
